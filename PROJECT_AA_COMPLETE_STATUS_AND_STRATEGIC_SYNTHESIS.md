### Google Business Profile Rules for Multiple Locations

Based on research from Search Engine Land's 2025 multi-location SEO guide:

#### **Option A: Separate GBP Listings Per Location** ✅ RECOMMENDED

| Requirement | Implementation |
|-------------|----------------|
| **Physical Address** | Each location must have unique, verifiable street address |
| **Phone Number** | Can share main practice phone OR use separate lines |
| **Verification** | Each location requires individual verification (postcard/phone) |
| **Website Linking** | Each GBP links to dedicated service area page on website |
| **Reviews** | Reviews managed per-location, not aggregated |

**Benefits:**
- ✅ Each office appears in local "near me" searches for its specific town
- ✅ Google Maps shows both locations when users search broadly
- ✅ Separate review management builds trust per location
- ✅ Clearer user experience (clients know which office to visit)

**Implementation:**
1. Claim existing GBP listings for both Ellijay and Blairsville
2. Verify each location via postcard or phone verification
3. Upload 20+ photos per location (office interior, treatment rooms, exterior)
4. Link Ellijay GBP to `/blue-ridge-ga` service area page
5. Link Blairsville GBP to `/blairsville-ga` service area page
6. Post weekly updates per location via GBP dashboard

#### **Option B: Single Service Area Business** ❌ NOT APPLICABLE

This option is only for businesses with NO physical offices (traveling practitioners). Since AA has two verified physical locations, this does not apply.

## 🎨 Technical Architecture Comparison

### Shared Stack (Both Sites)

| Component | Technology | Cost |

| **Framework** | Astro 5.x | Free |
| **Hosting** | Cloudflare Pages | Free |
| **Version Control** | GitHub | Free |

| **Scheduling** | Cal.com (acupuncture only) | Free tier available |
| **Total Monthly Cost** | - | **$0** |

### Key Technical Patterns Learned

#### 1. Centralized Configuration (`src/config/contact.ts`)
- Single source of truth for all contact info, pricing, business hours
- One edit updates ALL pages automatically
- Eliminates data duplication (15+ instances → 1 source)
- Easy for non-technical users to maintain

#### 2. Layout Content Injection (`<slot />` Not `{Astro.children}`)
- Critical Astro pattern learned through debugging
- Using React/Vue syntax causes blank pages
- Always use `<slot />` in layouts with `<html>`, `<head>`, `<body>` structure

#### 3. Global CSS Embedding
- When layout has full HTML structure, cannot import CSS at top
- Must embed all styles in `<style is:global>` block within Layout.astro
- Prevents CSS loading failures and blank pages

#### 4. JSON-LD Schema Injection
- Use `set:html` for injecting dynamic JSON into script tags
- Critical for LocalBusiness schema with geo-coordinates
- Ensures Google can parse location data correctly

## 🎓 Lessons Learned & Best Practices

### Technical Patterns That Worked:

1. **Centralized Configuration** (`src/config/contact.ts`)
   - Single source of truth for all business data
   - One edit updates entire site automatically
   - Eliminates data duplication and inconsistency risks

2. **Zero-JS Architecture (Astro)**
   - Perfect Core Web Vitals scores out of the box
   - Fastest possible load times = best SEO performance
   - No maintenance overhead for JavaScript dependencies

3. **Service Area Pages**
   - Dedicated pages for each town improve local rankings
   - Unique content per page avoids duplicate content penalties
   - Schema markup with geo-coordinates confirms location relevance

4. **Formspree Integration**
   - Simple, reliable email delivery without backend code
   - Free tier sufficient for current volume
   - Preserves AA's existing workflow (form → email → call)

### Patterns to Avoid:

1. ❌ Using `{Astro.children}` instead of `<slot />` in layouts
2. ❌ Importing CSS at top of full-page layouts (must embed with `<style is:global>`)
3. ❌ Near-duplicate service area pages (each must have unique content)
4. ❌ Missing GBP verification for multiple locations (each needs separate claim)
